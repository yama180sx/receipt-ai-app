# 指摘の優先度ルール（Must / Should / Later）

Issue #87 全体で共通。レビュア（人間・LLM）が迷ったときの判定基準です。

## Must（#87-2 / #87-3 / #87-4 の PR で対応）

次のいずれかに該当する場合のみ Must とする。**Epic 完了の必須条件**。

| 区分 | 例 |
|------|-----|
| **金額・精算の正しさ** | 按分合計 ≠ 明細小計、送金後 balance の符号ミス、端数メンバー以外に丸め残り |
| **データ整合** | Split 保存失敗時に UI だけ成功表示、トランザクション未使用で中途半端な DB 状態 |
| **認可・テナント** | `familyGroupId` 未検証で他世帯の Item / Transfer を更新 |
| **再現バグ** | 特定操作で必ず 500 / クラッシュ（ログ・手順が説明できる） |
| **型の穴で実行時障害** | 本番経路で `NaN` 表示、undefined 参照 |

**Must にしない例**

- 変数名・ファイル分割の好み
- 「サービス層に切り出すべき」だけの指摘（Should）
- テスト未整備（Later または #87-5 で手動チェックに記載）

## Should（Issue コメント or 新規 Issue 化。Epic 外でも可）

| 例 |
|-----|
| 重複した金額計算関数（動作は正しい） |
| `any` を具体型にできる（実行時は問題ない） |
| ログ・エラーメッセージの統一 |
| 軽微なパフォーマンス（体感なし） |

## Later（バックログ。Epic #246 の完了をブロックしない）

| 例 |
|-----|
| E2E テスト導入 |
| SplitEditor の全面コンポーネント分割 |
| 精算ロジックのドメインサービス層への完全移行 |
| UTC 月境界の厳密化（#87-0 以外で別検討） |

## Won't fix の記録

Must と判断したが直さない場合は、該当子 Issue に **理由・リスク受容** をコメントし、親 #246 からリンクする。

## 1 PR あたりの Must 上限

- 子 Issue 1本あたり **Must 3〜5件まで** を目安にする
- 超える場合は子 Issue を追加するか、Should に降格する
