# Issue #74: 起動時のタイムラグに関する技術調査および対策設計

## 1. 背景と課題
アプリ起動時において、メイン画面やログイン画面が表示されるまでに数秒〜十数秒のタイムラグ（白画面やActivityIndicatorの待機状態）が発生していた。本件はユーザー体感（UX）を著しく損なうため、ボトルネックの特定と対策を実施した。

## 2. ボトルネックの特定
調査の結果、遅延の原因は以下の3点に集約される。

1. **Expo開発モード（Development Mode）の特性**
   起動のたびにMetro Bundler（開発サーバー）から巨大なJavaScriptバンドル（ソースコード）をリアルタイムでダウンロード・同期するため、ネットワークおよびパースのラグが物理的に発生していた。
2. **ローカル永続化データのシーケンシャルロード**
   `AsyncStorage` および `SecureStore` からのデータ読み込み、およびトークン検証が直列（`await` の連続）で実行されており、I/O待ち時間が累積していた。
3. **初期化通信（API）のオーバーヘッド**
   ログイン状態であるにもかかわらず、カテゴリマスタ等の必要なデータを画面描画後に個別にフェッチしていたため、通信の往復遅延（RTT）分だけ描画後にロードインジケータが回り続ける原因となっていた。

## 3. 実施した対策（リファクタリング内容）
フロントエンド（`App.tsx`）のライフサイクルにおいて、以下の最適化を実施した。

### ① `expo-splash-screen` による初期ロードの隠蔽
ネイティブのスプラッシュ画面を明示的に維持（`preventAutoHideAsync`）し、すべてのデータ準備が整うまで裏側で処理を完結させる構成に転換。準備完了後にパッと画面を切り替えることで、ユーザーの「待たされている体感」を排除した。

### ② `Promise.all` によるストレージI/Oの完全並列化
トークン、メンバーID、前回表示ビュー、一時解析データのロードを `Promise.all` で一括実行し、非同期処理のオーバーヘッドを最小化した。

### ③ ログイン時マスタデータの先行フェッチ（投機的フェッチ）
トークンが存在（ログイン済み）する場合、スプラッシュ画面が閉じられる前にバックグラウンドで `/categories` を先行して取得。画面が描画された段階ですでにメモリ上にマスタが揃っている状態を作り出した。

## 4. 本番ビルド（Production/Release）時の挙動予測
本番ビルド（`eas build` またはローカルでのReleaseビルド）を適用することで、以下の効果が見込まれる。

* **JSバンドル同期の消失**: ソースコードがアプリバイナリ内にコンパイル・同梱されるため、起動時のサーバー通信が完全にゼロになり、数秒単位のラグがミリ秒単位へ劇的に短縮される。
* **アセットの静的化**: 画像やフォント等の読み込み順序に依存するチラつきが解消される。
